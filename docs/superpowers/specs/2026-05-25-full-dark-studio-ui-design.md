# Full Dark Studio UI Design

## Goal

为 pic_video_0515 前端建立一套统一的深色产品化界面，让项目页与编辑器共享同一视觉系统。

本次范围只覆盖前端界面结构与视觉表现，不改变后端接口契约，不重做节点功能逻辑，不引入新的业务流程。

## Approved Direction

用户已确认采用 full dark studio 方向，并额外明确两点：

- 节点必须保持深色，不能使用浅色卡片。
- 项目页面也必须是深色，不能与编辑器割裂成两个主题。

该方向的目标不是贴齐 testdir 的旧界面，而是在保持现有功能结构的前提下，让整体观感更接近产品化创作平台。

## In Scope

- 重设全局主题令牌与基础背景层。
- 重做项目页视觉结构，使其成为深色项目入口页。
- 重做编辑器外壳，包括顶部工具栏、左右侧栏与状态提示条。
- 统一节点外壳视觉语言，包括标题栏、选中态、运行态与标签层级。
- 保持现有页面路由、数据流、任务执行入口与节点组件边界不变。

## Out Of Scope

- 不新增页面。
- 不修改节点连接逻辑、任务执行逻辑或 API 请求逻辑。
- 不重构 Zustand store 结构。
- 不新增复杂动画库或额外 UI 框架。
- 不处理 backend parity 之外的接口问题。

## Files Expected To Change

核心实现文件应集中在以下已有入口：

- frontend/src/main.tsx
- frontend/src/styles/global.css
- frontend/src/pages/ProjectListPage.tsx
- frontend/src/pages/EditorPage.tsx
- frontend/src/components/flow/Toolbar.tsx
- frontend/src/components/flow/StatusBanner.tsx
- frontend/src/components/flow/nodes/shared/NodeShell.tsx

如果实现过程中发现必须触及相邻样式文件，可以扩展，但应保持改动集中，不散落到无关节点组件中。

## Design Rules

### 1. One Visual System

项目页与编辑器必须共享同一深色主题系统，用户从项目页进入编辑器时不能产生“切换到另一个产品”的感觉。

### 2. Dark Nodes Only

所有节点外壳保持深色表面。节点之间的区分只通过以下方式表达：

- 左侧窄色条
- 标题层级
- 标签与状态亮度差
- 选中或运行时的克制高光

不能通过浅色底卡来抬高节点。

### 3. Productized, Not Admin-Like

界面要更像创作工作台，而不是常规管理后台。实现上应避免以下表现：

- 通用后台式的平铺按钮带
- 大面积浅色内容底板
- 过多高饱和霓虹特效
- 密集表格感布局

### 4. Keep Existing Interaction Model

用户当前的使用习惯不能被破坏。按钮位置、主要入口和执行流程可以美化，但不应让用户重新学习完整操作路径。

## Page-Level Requirements

### Project List Page

目标：把当前项目列表页改成深色项目入口页。

要求：

- 页面整体使用深色背景与局部暖色强调。
- 顶部区域保留品牌标题、项目入口和 API 设置入口。
- 项目卡片继续承载“打开项目”的主行为。
- 项目卡片应具备更强的作品感或流程摘要感，而不是纯路径列表。
- 路径信息仍要可见，但降级为辅助层级。
- 空态也必须属于同一深色主题，不得落回默认浅色组件观感。

### Editor Page

目标：让编辑器成为统一的深色工作台。

要求：

- 保留当前三栏结构和主画布区域。
- 顶部工具栏改为更精炼的产品控制台风格。
- 左侧节点库、右侧帮助区、顶部状态条的背景层级必须统一。
- 背景网格与氛围效果可以保留，但需收敛到统一色系。
- 状态提示条不能再像高干扰通知，应更像嵌入式任务胶囊。

### Toolbar

目标：让工具栏看起来更高级，但不重排核心能力。

要求：

- 保留返回、保存、模板、历史、队列、执行、停止、帮助、API 等核心入口。
- 减少当前按钮风格混杂问题。
- 主动作与次动作必须有清晰层级。
- 移动端菜单逻辑保持可用，不改变其行为。

### Status Banner

目标：把任务状态收敛成更克制的界面部件。

要求：

- 保留运行中、全部完成、失败等状态表达。
- 视觉上从“提醒条”改为“嵌入式状态胶囊/状态卡”。
- 长任务描述必须可读，但不能破坏整体界面节奏。

### Node Shell

目标：统一所有节点外壳语言。

要求：

- 节点统一深色背景。
- 节点标题栏与内容区仍要有轻微层次区分。
- 选中态、执行态、成功态、失败态都需要可感知，但要避免旧版强霓虹效果。
- 高性能模式下仍允许弱化复杂阴影或动效。
- 不改变节点组件对 NodeShell 的调用方式。

## Theme Tokens

实现时应优先收敛为一套新的深色主题变量，至少覆盖：

- 页面背景色
- 面板背景色
- 节点背景色
- 边框色
- 主文本色
- 次文本色
- 强调色
- 状态色

这些令牌应优先落在全局主题层和 global.css，而不是散落在每个组件的行内样式中。

## Motion And Effects

只允许少量有目的的动效：

- 页面初始加载的轻量层次动效
- 节点选中或执行的克制高光
- 按钮 hover 的小幅反馈

不应引入大面积持续闪烁、频繁呼吸光效或视觉噪音较高的动画。

## Accessibility And Usability

- 深色主题下文字对比度必须保持可读。
- 主要按钮与危险按钮在视觉上必须可区分。
- 项目页和编辑器页都要兼顾桌面端与窄宽度场景。
- 现有按钮文本继续使用中文。

## Validation Requirements

正式实现阶段至少需要以下验证：

- frontend 构建通过
- 受影响文件静态错误扫描通过
- 如有样式或结构改动导致交互异常，需做一次针对项目页与编辑器页的 focused validation

## Implementation Constraints For Next Step

- 先从全局主题与外壳层入手，再收口节点外壳。
- 避免一次性重写所有节点组件。
- 项目页与编辑器壳体可以先实现，再微调局部组件细节。
- 如果实现过程中发现 mockup 与现有组件结构冲突，优先保留现有结构并实现最接近的视觉结果，而不是为了视觉稿强行重构数据层。

## Non-Goals To Protect Scope

以下内容即使在实现过程中看起来相关，也不应自动并入本轮：

- 新的 onboarding 页面
- 新的模板系统信息架构
- 节点参数布局重构到全新交互模型
- 后端接口联调扩展
- Electron 主进程行为调整

## Ready-For-Planning Decision

该规格已经明确：

- 要改哪些页面和组件
- 哪些视觉原则不能违反
- 哪些功能边界不能动
- 后续实现时的验证标准

因此下一步应进入实现计划，而不是继续扩大设计范围。
